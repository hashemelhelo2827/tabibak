1. Original State — Local Only

The application was a single tabibak.html file (~4300 lines) with embedded CSS and JavaScript. All data was stored in the browser's localStorage:

FeatureOriginal Implementation
Data StoragelocalStorage (browser-only, single device)
User AuthPlain text passwords in localStorage, no hashing
APINo backend — all logic in the browser
Real-time SyncNot supported
DeploymentOpen HTML file directly from desktop (file://)
Data PersistenceCleared on browser cache reset, single device only
Limitations:
• Data not shared across devices
• Lost on browser cache clear
• No user authentication security (plain text passwords)
• No backend for push notifications or scheduled reminders

The architecture was transformed from a single HTML file to a full-stack client-server application:

BEFORE: Browser ↔ localStorage

AFTER:
Browser (HTML/CSS/JS) → Node.js/Express API → Turso (Cloud SQLite)
Browser ← Socket.io (Real-time) ← Node.js/Express API
LayerTechnologyPurpose
FrontendHTML5, CSS3, Vanilla JSUI (same as original, with API integration)
BackendNode.js + ExpressREST API server on port 3000
Real-timeSocket.io (v4)WebSocket events for cross-device sync

HostingRailway (SFO region)Auto-scaling Node.js hosting
Authbcrypt + JWTPassword hashing + token-based sessions

4. Database — From localStorage to Turso Cloud SQLite

Step 1 Replace better-sqlite3 with @libsql/client
The synchronous SQLite driver was replaced with Turso's async client, which works with both local files and cloud databases.

Step 2 Same SQL Schema
All tables use standard SQLite syntax — no migration needed:

TablePurpose
usersUsername, bcrypt password hash, profile, FCM token
sessionsTriage assessment history with full results (JSON)
medicationsMedication name, dose, form, urgency flag
medication_dosesDose times per medication (one-to-many)
medication_logDaily dose taken tracking

Step 3 Turso Cloud Database Created
Database URL: libsql://tabibak-hashemelhelo2827.aws-ap-south-1.turso.io
Free tier: 500 databases, 9 GB storage, 1B row reads/month

Fallback behavior: When TURSO_DB_URL + TURSO_DB_TOKEN are not set, the app falls back to a local SQLite file — keeping local development working without changes.

6. Real-Time Synchronization with Socket.io

WebSocket events are emitted whenever data changes, enabling instant cross-device sync:

EventTriggered WhenWhat Clients Do
medications:updatedMedication created or deletedRefetch & re-render meds list
medications:dose-loggedDose marked as takenRefetch meds + update progress
sessions:createdTriage session savedRefetch & refresh history screen

Flow:

Phone: Add Medication → POST /api/medications → Server saves to Turso
Server emits medications:updated → room:user:john
Laptop (in room:user:john) receives event → auto-refetches data → updates UI
→ No page refresh needed
